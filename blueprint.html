💡
Por que sem registry externo? O Coolify pode fazer o build do Dockerfile diretamente no servidor a partir do repositório Git — sem precisar empurrar imagem para o Docker Hub ou GitHub Container Registry. Isso simplifica o pipeline, reduz custos e elimina dependências externas no fluxo de deploy. A imagem existe apenas localmente no servidor.

08 — Segurança
Defesa em Camadas
Múltiplas camadas de proteção desde a borda até o container. Cada item abaixo foi verificado no código-fonte do template.
🌐
Cloudflare — Edge Protection
DDoS mitigation, WAF, bot management e SSL termination antes de chegar ao servidor. IP real do servidor oculto atrás do proxy Cloudflare.
DDoS mitigationWAFSSL edgeIP hiding
🔥
Firewall — Hetzner + UFW (dual-layer)
Hetzner Cloud Firewall como primeira camada + UFW no SO. Apenas portas 22, 80, 443 e 8000 (Coolify UI). Deny-all por padrão.
deny-all defaultport allowlistdual-layer
🔑
SSH Hardening
Porta não-padrão (2222), apenas chave ED25519, sem root login, AllowUsers restrito. Fail2Ban com ban de 24h após 3 tentativas falhas.
ED25519 onlyno password authFail2Banporta 2222
🍪
Autenticação BFF + httpOnly Cookies
Tokens em cookies httpOnly + Secure + SameSite. Frontend nunca acessa o token — BFF pattern via auth-bff.controller.ts. Refresh tokens criptografados com AES-256-GCM no banco.
httpOnly: truesecure: trueBFF patternAES-256-GCM
🛡️
Headers HTTP — Helmet.js
Configurado em main.ts com HSTS (1 ano + preload), X-Frame-Options: DENY, X-Content-Type-Options: nosniff, CSP com diretivas restritas, Referrer-Policy strict-origin-when-cross-origin.
HSTS preloadX-Frame DENYCSPnosniff
⏱️
Rate Limiting — 3 camadas
ThrottlerModule com 3 tiers: short (3/s), medium (20/10s), long (100/min). Decorators customizados: @StrictThrottle (auth), @ApiThrottle (API), @AuditThrottle (operações caras). Tracking por user ID ou IP.
3 req/s20 req/10s100 req/minper-user tracking
🔒
CORS — Origins explícitos
CORS em main.ts com callback dinâmico — origins carregados via CORS_ORIGIN (env var). Localhost só em development. Credentials: true para cookies. Nunca usa origin: '*'.
env-driven originscredentials: trueno wildcard
✅
Validação — class-validator + ValidationPipe
ValidationPipe global com whitelist: true, forbidNonWhitelisted: true, transform: true. Todos os DTOs usam decorators: @IsString, @MaxLength, @IsUUID, @ValidateNested. Requests com campos extras retornam 400.
whitelistforbidNonWhitelistedtransform
🐳
Container Security
Multi-stage build (Alpine). Usuário non-root nestjs:1001. Tini como init (PID 1). .dockerignore exclui .env, .git, specs. DB e Redis sem porta pública — rede interna Docker.
non-rootmulti-stagetini init.env excluded
📋
Logs — Sanitização automática
Winston com JSON estruturado em produção. AuditLoggerInterceptor em todas as requests com requestId (UUID). Lista de campos sensíveis (password, token, apiKey, creditCard, cpf) redacted para [REDACTED] automaticamente.
JSON structuredauto-redactcorrelation IDaudit trail
09 — Observabilidade
Monitoramento & Alertas
Stack completa provisionada como código. Todos os serviços no docker-compose.observability.yml, rede isolada, sem portas públicas.
📊
Grafana 11.5
Dashboards provisionados como código (JSON). 2 dashboards pré-configurados: NestJS API (request rate, latency p50/p95/p99, error rate, memory, event loop) e Logs (volume por level, error logs, live tail). Datasources Prometheus + Loki auto-provisionados.
dashboards as codedatasources auto2 dashboards
🔥
Prometheus 2.53
Scrape a cada 10s do endpoint /metrics do NestJS (prom-client). Métricas: request rate, duration histogram, in-flight, response size, Node.js heap, event loop lag, CPU, active handles. Retenção: 30 dias.
/metrics endpointprom-client10s scrape30d retention
📝
Loki 3.4 + Promtail
Promtail coleta logs de todos os containers Docker via socket. Pipeline: parsing JSON, extração de level/timestamp/context, labels automáticos (container, service). Winston JSON em produção. Retenção: 30 dias. Consulta via LogQL no Grafana.
PromtailJSON pipelineLogQL30d retention
🚨
Alertas Provisionados
6 alertas configurados como código: error rate > 1% (critical), p95 latency > 2s, RSS memory > 512MB, event loop lag > 100ms, target down (critical). + alerta de volume de error logs > 10/min via Loki.
6 alert rulesas codeTelegram/Slack
📡
Uptime Kuma 1.23
Container dedicado para status page e monitoramento de uptime. Checks HTTP a cada 60s nos endpoints: /health, /health/ready, /health/live. Alertas por Telegram/Slack quando serviço cai.
status page60s checksTelegram alerts
💾
Backup PostgreSQL → S3
Container pg-backup com cron a cada 6h. pg_dump comprimido (gzip), upload para S3 (STANDARD_IA). Retenção local: 7 dias. Script em monitoring/backup/pg-backup-s3.sh. Env vars: S3_BACKUP_BUCKET, DATABASE_URL, AWS credentials.
6h crongzip + S3STANDARD_IA7d local
